# Claude Code プロジェクトガイド

## 基本ルール

- **応答言語**: すべての応答は日本語で行うこと
- **絵文字禁止**: ドキュメントおよび応答に絵文字を使用しないこと
- **目次の記載**: マークダウンファイルには目次を記載すること（番号なし、リンク形式）
- **設計補助ファイルの命名規則**: 設計の補助ファイル・ディレクトリには `_` プレフィックスをつけること

目次の形式:
```markdown
## 目次

- [セクション名](#セクション名)
- [セクション名](#セクション名)
```

設計補助ファイルの例:
- `_templates/` - ドキュメント作成テンプレート
- `_interview/` - ヒアリング用質問・回答
- `_business-requirements.md` - ビジネス要件のテンプレート

## プロジェクト概要

このレポジトリは、Claude Codeを使用した仕様駆動開発のためのガイドラインとテンプレートを提供します。
新規プロジェクト開始時にこのレポジトリをコピーし、ガイドラインとテンプレートをもとにプロジェクト固有のファイルを作成して使用します。

## ドキュメント構造

`docs/` ディレクトリ配下に、開発フェーズに沿った体系的なドキュメントを配置しています。

| ディレクトリ | フェーズ | 内容 |
|-------------|---------|------|
| `00-rules/` | （横断） | 開発ルール・コーディング規約 |
| `01-planning/` | 企画・要件定義 | ビジネス要件、機能要件、非機能要件、ユースケース |
| `02-basic-design/` | 基本設計 | アーキテクチャ、UI設計、API設計、DB設計、セキュリティ |
| `03-detail-design/` | 詳細設計 | フロントエンド/バックエンド処理ロジック、シーケンス図 |
| `04-testing/` | テスト設計 | テスト計画、単体/結合/E2Eテスト仕様 |
| `05-operations/` | 運用設計 | 監視、運用手順、障害対応、リリース手順 |

詳細な構成は `docs/README.md` を参照してください。

## 開発時の指針

### コード作成・修正時

1. **必ず関連ドキュメントを確認**
   - `docs/00-rules/` のコーディング規約に従う
   - `docs/01-planning/` で要件を確認
   - `docs/02-basic-design/` でUI・API・DB仕様を確認
   - `docs/03-detail-design/` で処理ロジックを確認

2. **設計ドキュメントの役割分担**
   - 基本設計（02-basic-design）: 「何を」作るか（インターフェース定義）
   - 詳細設計（03-detail-design）: 「どのように」実装するか（処理ロジック）

3. **テストの実装**
   - `docs/04-testing/` のテスト戦略に従う
   - 必要なテストケースをカバー

### ドキュメント参照の優先順位

コードを書く前に、以下の順序でドキュメントを確認してください：

1. `00-rules/` - プロジェクト全体のルールを理解
2. `01-planning/` - 実装する機能の要件を確認
3. `02-basic-design/` - 基本設計の仕様を確認
4. `03-detail-design/` - 詳細設計（処理ロジック）を確認
5. `04-testing/` - テスト要件を確認

## 作業フロー

### 新機能実装時

1. `docs/01-planning/` で要件を確認
2. `docs/02-basic-design/` で基本設計（UI・API・DB）を確認
3. `docs/03-detail-design/` で詳細設計を確認
4. `docs/00-rules/` のコーディング規約を遵守して実装
5. `docs/04-testing/` に基づいてテストを作成
6. 実装とドキュメントの整合性を確認

### バグ修正時

1. 関連する機能の `docs/02-basic-design/` と `docs/03-detail-design/` を確認
2. `docs/00-rules/` のコーディング規約を遵守して修正
3. `docs/04-testing/` に基づいて回帰テストを実施

### リファクタリング時

1. `docs/02-basic-design/architecture/` の設計方針を確認
2. `docs/00-rules/` のコーディング規約に従う
3. 既存の仕様との整合性を保つ
4. テストが通ることを確認

## 注意事項

- ドキュメントが存在しない場合、まずドキュメントの作成を検討してください
- 実装とドキュメントに乖離がある場合、必ずドキュメントを更新してください
- 不明点がある場合、推測せずにユーザーに確認してください
- このレポジトリはテンプレートのため、空のディレクトリが存在する可能性があります
- **Git管理について**: すべての変更履歴はGitのコミット履歴で管理されます。ドキュメント内にバージョン履歴を記載する必要はありません
